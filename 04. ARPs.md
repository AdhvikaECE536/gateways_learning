ARP stands for **Address Resolution Protocol**
- has to find MAC address that belongs to IP address on local network
- when laptop wants to send packet to 192.168.1.1 (router), IP layer knows the destination IP. But ethernet- the actual wire- needs a MAC address to deliver the frame. Bith the systems dont know each other, ARP is the bridge.
- between L2 and L3; RFC 826 (1982) defines it
- only works within a single braodcast domain (local subnet); ARP never crosses a router

Computer works with two completely different address systems simultaneously-
- IP address:
    - are 32 bit logical; assigned by the software
    - used by routers to forward packets between networks
- MAC address:
    - are 48 bit physical; burned into hardware of every network card
    - used by switches to deliver frames within a local network
 
# How it works?

Scenario: Your laptop (192.168.1.50) wants to send a packet to your router (192.168.1.1). It knows the IP but not the MAC.

1
Check the ARP cache first
Before sending anything, your OS checks its local ARP cache — a table of recently resolved IP→MAC mappings. If 192.168.1.1 is already there, done. No ARP needed. If not, proceed to step 2.

2
Broadcast an ARP Request
Your laptop shouts to the entire local network: "Who has IP 192.168.1.1? Tell 192.168.1.50." This is sent as an Ethernet broadcast — destination MAC is FF:FF:FF:FF:FF:FF, meaning every device on the subnet receives it. Every single device on the LAN gets this frame.

3
Every device reads the request
All devices on the subnet (your phone, your smart TV, every laptop, the printer) receive the broadcast. Each one reads the target IP. If it doesn't match their IP, they silently discard the frame. No reply, no action.

4
The target replies with a unicast ARP Reply
The router recognises its own IP (192.168.1.1) in the request. It sends an ARP Reply — but critically, this is unicast, not broadcast. It goes directly back to your laptop's MAC address: "192.168.1.1 is at AA:BB:CC:11:22:33."

5
Your laptop caches the result
Your OS stores the mapping 192.168.1.1 → AA:BB:CC:11:22:33 in the ARP cache with a TTL (typically 20 minutes on Linux/Windows). Future packets to that IP skip ARP entirely and use the cached MAC directly.

6
Original packet is sent
Now that the MAC is known, the original IP packet is wrapped in an Ethernet frame addressed to AA:BB:CC:11:22:33 and sent. The router receives it, reads the IP header, and routes it onward.


A device can send an ARP Request for its own IP address. This is called a **gratuitous ARP**. It's used to: (1) announce itself to the network after getting a new IP, (2) detect IP conflicts (if someone replies, two devices share an IP), and (3) update other devices' ARP caches when a MAC address changes (e.g. after a failover). The reply is optional — no one needs to respond.

**ARP Reply differences**

In the Reply, OPER = 0x0002. The sender/target fields are swapped — the router fills in its own MAC as the sender, puts the laptop's MAC as target, and the THA (target hardware address) in the original request is now filled with the router's actual MAC. The Ethernet dest MAC in the reply is unicast to the requester's MAC — not broadcast.
